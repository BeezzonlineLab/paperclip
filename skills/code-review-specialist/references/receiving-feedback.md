# Receiving Code Review Feedback

When you receive review feedback on your code:

## Rules
1. **Read fully before reacting** — understand the entire review before responding
2. **Assume good intent** — the reviewer wants to improve the code, not criticize you
3. **Distinguish categories** — critical (must fix), major (should fix), minor (optional)
4. **Fix criticals immediately** — no debate, these block merge
5. **Discuss majors if you disagree** — provide reasoning, not defensiveness
6. **Accept or ignore minors** — your call, but acknowledge you read them

## Response Protocol
- **If you agree:** Fix it, commit, comment "Fixed in [commit]"
- **If you disagree:** Explain your reasoning with specific technical justification
- **If you need clarification:** Ask a specific question, not "what do you mean?"
- **If the reviewer is wrong:** Be respectful, cite documentation or examples

## Anti-Patterns
- ❌ "It works though" (correctness ≠ quality)
- ❌ "That's how it was before" (legacy is not justification)
- ❌ "I'll fix it in a follow-up" (unless truly minor)
- ❌ Ignoring feedback without acknowledging it
